# 03. チューニング

CPUオフロード(または swap への逃避)を最優先で回避し、長コンテキスト運用するための調整指針。

> 本書は **Ollama / mlx-lm の両方** で共通の方法論。コマンド例は Ollama を主にするが、Mac/MLX 環境では「VRAM → ユニファイドメモリの `max_recommended_working_set`」、「`num_ctx` → `max_kv_size`」、「`OLLAMA_KV_CACHE_TYPE` → `--kv-bits`」と読み替えればそのまま使える。

## 1. メモリ消費の構造

GPU(または Apple Silicon GPU)に乗るものは大きく3つに分かれます。

```
GPU メモリ消費 = モデル重み + KVキャッシュ + 推論バッファ(その他)
```

| 要素         | 主に効くもの                             | 削減手段                                                 |
| ------------ | ---------------------------------------- | -------------------------------------------------------- |
| モデル重み   | パラメータ数・量子化レベル               | より小さなモデルへ / より強い量子化(Q5→Q4)               |
| KVキャッシュ | コンテキスト長・モデルのアテンション構造 | `num_ctx` / `max_kv_size` を下げる / KV キャッシュ量子化 |
| 推論バッファ | バッチサイズ・並行リクエスト数           | Ollama: `OLLAMA_NUM_PARALLEL=1` / MLX: 並行未サポート    |

> **Mac/MLX 環境では「VRAM」はユニファイドメモリの一部**。`max_recommended_working_set_size`(`mx.device_info()` で取得)を「実効 VRAM」相当として扱い、超過すると macOS が swap に逃がして全体が遅くなる。

## 2. モデル重みの量子化

GGUF系の代表的な量子化レベルの目安。Ollamaのデフォルトタグは多くが `Q4_K_M` です。

| レベル   | サイズ目安(7B 基準) | 品質劣化       | 用途                   |
| -------- | ------------------- | -------------- | ---------------------- |
| `Q8_0`   | 約 7.0 GB           | ほぼ無視できる | 品質優先               |
| `Q6_K`   | 約 5.5 GB           | わずか         | 品質寄り               |
| `Q5_K_M` | 約 4.8 GB           | 小             | バランス               |
| `Q4_K_M` | 約 4.0 GB           | 小〜中         | **一般的な実用最適点** |
| `Q4_0`   | 約 3.8 GB           | 中             | 速度重視               |
| `Q3_K_M` | 約 3.3 GB           | 中〜大         | ギリギリ収めたい時     |
| `Q2_K`   | 約 2.8 GB           | 大             | 緊急回避               |

経験則: **Q4_K_M を起点に、VRAM 不足なら Q4_0 か Q3_K_M、品質を上げたければ Q5_K_M か Q6_K へ**。

## 3. KVキャッシュ量子化

長コンテキスト運用では KV キャッシュが支配的になります。

**Ollama (`OLLAMA_KV_CACHE_TYPE`)** の主要値:

| 値              | bytes/element | サイズ感 | 影響                               |
| --------------- | ------------- | -------- | ---------------------------------- |
| `f16` (default) | 2             | 100%     | 基準                               |
| `q8_0`          | ~1.06         | 約 53%   | 品質劣化はほぼ感じられない         |
| `q4_0`          | ~0.56         | 約 28%   | 長文の参照精度がやや落ちる場合あり |

**前提**: KVキャッシュ量子化を効かせるには `OLLAMA_FLASH_ATTENTION=1` が必要。

**mlx-lm (`--kv-bits`)** の主要値:

| 値             | bits | サイズ感 | 影響                                       |
| -------------- | ---- | -------- | ------------------------------------------ |
| 未指定 (既定)  | f16  | 100%     | 基準                                       |
| `--kv-bits 8`  | 8    | 約 50%   | 品質劣化はほぼ感じられない                 |
| `--kv-bits 4`  | 4    | 約 25%   | 長文の参照精度がやや落ちる場合あり         |

mlx-lm の KV 量子化はリクエスト時引数で切り替わるため、サービス再起動は不要。`kv_group_size=64` (既定) も併せて指定可能。`quantized_kv_start` で「最初の N トークンは非量子化のまま」という挙動も指定できる(本評価では既定の 0 で固定)。

## 4. KVキャッシュサイズの概算

GQA(Grouped Query Attention)ありのモデルでの 1 トークンあたり KVキャッシュサイズ:

```
KV/token = 2 × num_layers × num_kv_heads × head_dim × bytes_per_element
```

- 係数 `2` は Key と Value の両方を持つため
- `bytes_per_element` は f16=2, q8_0≈1.06, q4_0≈0.56

長コンテキスト時の総KV:

```
KV total = (KV/token) × num_ctx
```

モデル仕様(`num_layers`, `num_kv_heads`, `head_dim`)はモデルカードに記載されています。Hugging Face の `config.json` でも確認可能。

**目安**: GQA採用の中型モデル(2B〜4B級)で f16・128k → 4〜8 GB、q4_0 にすると 1〜2 GB まで縮みます。

## 5. Flash Attention

| GPU世代            | FA有効化の効果                                                                         |
| ------------------ | -------------------------------------------------------------------------------------- |
| Pascal(CC 6.x)     | 動作はする。速度メリットは限定的。KVキャッシュ量子化を効かせる前提条件として有効化する |
| Turing(CC 7.5)     | 速度・VRAM双方で恩恵あり                                                               |
| Ampere(CC 8.x)以降 | 本領発揮。常時有効推奨                                                                 |

`OLLAMA_FLASH_ATTENTION=1` を入れた後、起動ログを必ず確認:

```bash
journalctl -u ollama -n 100 --no-pager | grep -iE "flash|attention"
```

「flash attention enabled」のような表示が出ていればOK。出ていない/警告が出ている場合はサポート外の可能性。

## 6. 調整フロー

### Ollama (Linux)

目標 `num_ctx` でロードしてみて `PROCESSOR` を見るところから始めます。

```
┌─ ロードして ollama ps を見る ─┐
│
├─ PROCESSOR が 100% GPU
│    └─ 完了。実用速度が出ているなら採用、低ければ 8 へ
│
├─ PROCESSOR に CPU% が含まれる
│    ├─ 1. OLLAMA_FLASH_ATTENTION=1 を有効化 → 再計測
│    ├─ 2. OLLAMA_KV_CACHE_TYPE=q8_0 に変更 → 再計測
│    ├─ 3. OLLAMA_KV_CACHE_TYPE=q4_0 に変更 → 再計測
│    ├─ 4. num_ctx を半分に縮めて再計測(128k→64k→32k)
│    ├─ 5. より強い重み量子化(Q4_K_M → Q4_0 → Q3_K_M)
│    ├─ 6. より小さいモデルに変更
│    └─ 7. それでもダメなら、当該GPUでは目標 ctx 不可と判定
│
└─ そもそもロードが失敗する(OOM)
     └─ 上記 4〜6 を先に適用してリトライ
```

### mlx-lm (macOS)

目標 `max_kv_size` で `run_speed_mlx.py` を回してピークと swap を見るところから始めます。

```
┌─ run_speed_mlx で peak_memory_mib を取る ─┐
│
├─ peak ≤ effective_gpu_limit - safety_margin
│    └─ 完了。実用速度が出ているなら採用
│
├─ peak が limit を超過 / swap 発生 / 例外
│    ├─ 1. --kv-bits 8 を指定 → 再計測
│    ├─ 2. --kv-bits 4 を指定 → 再計測
│    ├─ 3. --ctx (max_kv_size) を半分に縮めて再計測
│    ├─ 4. より強い重み量子化のタグへ(8bit → 4bit、または OptiQ-4bit)
│    ├─ 5. より小さい/軽いモデルに変更
│    ├─ 6. iogpu.wired_limit_mb を引き上げ(副作用注意)
│    └─ 7. それでもダメなら、当該 Mac では目標 ctx 不可と判定
│
└─ そもそもロードが失敗する
     └─ 重み単体が working_set を超えている。4 または 5 を先に適用
```

## 7. 量子化のトレードオフ

| 観点                       | 強い量子化(Q4_0, q4_0)                                      | 弱い量子化(Q6_K, q8_0 / f16) |
| -------------------------- | ----------------------------------------------------------- | ---------------------------- |
| VRAM                       | 小                                                          | 大                           |
| 速度                       | 速い(メモリ帯域節約)                                        | やや遅い                     |
| 出力品質                   | 劣化あり(特に長文の整合性、コード生成)                      | 高                           |
| KVキャッシュ q4_0 の副作用 | 長コンテキスト内の **過去情報の参照精度**が落ちることがある | なし                         |

長文の参照タスク(RAG、コードベース理解など)で精度が要るなら、コンテキストを短くしてでも KVキャッシュは `q8_0` 以上を維持する判断もあり。

## 8. 速度がさらに欲しい場合

- **モデルサイズを下げる**: 4B → 2B で約2倍速くなることが多い
- **GGUF量子化を下げる**: Q5_K_M → Q4_0 でメモリ帯域節約により高速化
- **`num_predict` を必要な長さに制限**: 不要なトークン生成を避ける
- **GPU側の電力リミット見直し**: `nvidia-smi -pl <watts>` で上限を解放(冷却が前提)
- **PCIe帯域不足の疑い**: 巨大プロンプト時にCPU↔GPU転送がボトルネックになる場合あり

## 9. 再現性の落とし穴

- `OLLAMA_NUM_PARALLEL` を変えると KV サイズが変わり、同じ `num_ctx` でも VRAM が違う
- Ollama のバージョン更新でデフォルト値や FA の挙動が変わる場合がある(再計測推奨)
- 別プロセス(Xorg、別CUDAアプリ)が VRAM を食っていないか毎回確認
- `nvidia-smi --query-compute-apps=pid,used_memory --format=csv` で常駐プロセスを把握
